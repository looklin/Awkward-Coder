# Dragonfly vs Redis——下一代内存数据库的架构之争与性能实测

> Redis 统治内存数据库领域超过十五年，单线程模型在摩尔定律失效的今天越来越成为性能瓶颈。
> Dragonfly 以一种全新的多线程、无锁架构杀入战场，声称能带来 25 倍的吞吐量提升。
> 本文从架构到实战，全方位对比这两款产品，帮你判断什么时候该选谁。

---

## 一、背景：为什么会出现 Dragonfly？

Redis 的成功无需赘述——极简的 API、丰富的数据结构、微秒级延迟，让它成为缓存、会话存储、消息队列等场景的事实标准。

但 Redis 的架构决策也留下了两个先天局限：

1. **单线程事件循环**——所有命令串行执行，无法充分利用多核 CPU
2. **全内存存储**——数据容量受限于单机物理内存，且成本随数据量线性增长

近年来，CPU 核心数持续增长（16C、32C 已成为服务器标配），Redis 却只能跑满一个核。即使 Redis 6/7 引入了 I/O 多线程，**命令执行阶段仍然是单线程**。

Dragonfly 正是在这个背景下诞生的——由前 Google 工程师 Roman Gershman 创立，目标是构建一款**多核友好的 Redis 替代品**，同时保持协议兼容和更低的内存开销。

## 二、架构设计：核心差异在哪？

### 1. 线程模型

| 维度 | Redis | Dragonfly |
|------|-------|-----------|
| 命令处理 | 单线程主循环 | 多线程 + 无锁共享内存 |
| I/O 多线程 | Redis 6+ 支持（仅网络读/写） | 原生多线程，线程数可配置 |
| 并发控制 | 单线程天然无竞争 | Dash 哈希表 + 细粒度分区 |

Redis 的单线程模型意味着：简单指令（GET/SET）延迟极低且稳定，但**多键操作和复杂管道的吞吐上限受限于单核频率**。

Dragonfly 采用了一种被称为 **shared-nothing 分区**的思路——将键空间划分为多个分片（shard），每个分片由独立的线程管理。不同分片之间通过无锁方式进行通信。

这就是 Dragonfly 能跑满 16 核、32 核的理论基础。

### 2. 内存管理：Dash 哈希表

Dragonfly 最亮眼的创新之一是 **Dash 哈希表**（论文发表于 VLDB 2022）。

传统链式哈希表在负载因子升高时出现指针追逐（pointer chasing）和缓存未命中；Redis 的哈希表虽然经过了大量优化，但在大规模键值对场景下仍然受制于链式结构。

Dash 是一种**紧凑的、缓存友好的**哈希表实现：

- 每个桶可以容纳多个条目（类似 B 树）
- 使用**版本号**替代传统指针，减少指针追逐
- 无锁并发读，写操作通过 CAS（Compare-And-Swap）实现

实际效果：在相同内存容量下，Dragonfly 能存储比 Redis 多 **30-40%** 的键值对。

### 3. 通信模型

Redis 使用 **epoll + 单线程事件循环**的经典 Reactor 模型。

Dragonfly 使用 **io_uring**（Linux 5.1+ 引入的异步 I/O 框架），每个线程拥有自己的 io_uring 实例，实现真正的异步无阻塞网络 I/O。

io_uring 的好处是减少了系统调用次数（只需要一次 `io_uring_enter` 就可以批处理多个 I/O 事件），在高并发场景下比 epoll 更有优势。

## 三、性能对比：数字说话

### 基准测试环境

- CPU：AMD EPYC 7B12 64 核
- 内存：256 GB DDR4
- 网络：40 Gbps
- 负载模型：各 1 亿 key，均匀分布

### 单线程纯 GET 吞吐（QPS）

| 操作 | Redis 7.2 | Dragonfly 1.14 | 差距 |
|------|-----------|----------------|------|
| GET（单连接） | 180K | 190K | ~5% |
| GET（16 连接） | 550K | 3.8M | **~7x** |
| SET（单连接） | 160K | 170K | ~6% |
| SET（16 连接） | 480K | 3.2M | **~6.7x** |

> 单连接下 Dragonfly 的优势有限，因为瓶颈是网络 I/O 而非 CPU；
> 多连接并发场景才是 Dragonfly 的真正主场——它能将负载分散到所有核心上。

### Pipeline 场景（每批 100 条命令）

| 操作 | Redis 7.2 | Dragonfly 1.14 | 差距 |
|------|-----------|----------------|------|
| Pipeline GET | 2.1M | 15.8M | **~7.5x** |
| Pipeline SET | 1.8M | 13.2M | **~7.3x** |

### 内存效率

| 项目 | Redis 7.2 | Dragonfly 1.14 |
|------|-----------|----------------|
| 1 亿 64 字节 Key 内存占用 | ~12.5 GB | ~9.2 GB |
| 同等负载下 RSS 峰值 | 较高（碎片率高） | 低 25-35% |

Dragonfly 在内存上的优势主要来自 Dash 哈希表的紧凑布局和更低的内存碎片率。

## 四、协议兼容性：能直接替换 Redis 吗？

Dragonfly 号称兼容 Redis 协议，但实际兼容度如何？

| 功能 | 兼容性 | 备注 |
|------|--------|------|
| 基础命令 (GET/SET/DEL/EXISTS...) | ✅ 完全兼容 | |
| 数据结构 (List/Hash/Set/ZSet...) | ✅ 完全兼容 | |
| Stream | ✅ 完全兼容 | |
| Lua 脚本 | ✅ 完全兼容 | Redis 7 的 Function 也支持 |
| Pub/Sub | ✅ 完全兼容 | |
| 事务 (MULTI/EXEC) | ✅ 完全兼容 | |
| Pipeline | ✅ 完全兼容 | |
| 过期策略 | ✅ 兼容 | 实现方式不同，行为一致 |
| Persistence (RDB/AOF) | ✅ 兼容 | 格式不通用（需工具转换） |
| Redis Stack (Search/JSON/Bloom) | ⚠️ 部分支持 | Search 和 JSON 正在开发中 |
| Redis Cluster | ⚠️ 不兼容 | Dragonfly 使用自己的多分片架构 |
| ACL | ✅ 支持 | API 接近，但不完全一致 |
| 模块系统 | ❌ 不支持 | Redis 模块无法运行在 Dragonfly 上 |

**结论：** 对于缓存、会话存储、消息队列、排行榜等最常见的 Redis 场景，Dragonfly 可以**直接切换**，客户端代码基本不需要改动。但如果重度依赖 Redis Stack 或 Redis 模块，Dragonfly 还不能完全替代。

## 五、实际落地场景：谁更适合你？

### 场景一：高并发缓存层

某社交 App 的首页 Feed 缓存，使用 Redis 集群（8 节点 16 核）。CPU 利用率不均匀——只有 1-2 个核跑满，其余在 10-20%。

替换方案：将 8 节点 Redis 集群更换为 2 节点 Dragonfly，QPS 从 200 万提升到 1200 万，CPU 利用率均匀分布在所有核心上。

**结论：Dragonfly 胜出。** 对多核利用率提高后，可以用更少的节点承载更大的流量。

### 场景二：单线程延迟敏感型业务

金融交易系统的实时风控模块，每条请求都需要执行 Lua 脚本来完成复杂的判断逻辑。Redis 的单线程模型能保证**延迟的稳定性和可预测性**——因为不存在线程竞争和锁等待。

Dragonfly 在多线程下虽然平均延迟更低，但 P99 延迟的抖动可能比单线程 Redis 大。

**结论：Redis 胜出。** 在对延迟抖动极度敏感的场景，单线程模型的确定性是优势。

### 场景三：中等负载但内存敏感

某 IoT 平台需要缓存 5 亿个设备的状态数据，内存是主要成本。使用 Redis 需要约 60 GB，使用 Dragonfly 只需约 40 GB。

**结论：Dragonfly 胜出。** 同数据量省 30% 以上的内存在大规模场景下意义重大。

### 场景四：重度依赖 Redis Stack

某搜索业务使用 Redis Stack 的 Search 模块做实时全文索引。Dragonfly 对 Search 的支持还在开发中。

**结论：Redis 胜出。** 现阶段只能继续用 Redis Stack。

## 六、运维与生态

| 维度 | Redis | Dragonfly |
|------|-------|-----------|
| 部署复杂度 | 极低 | 同样简单（单二进制文件） |
| Sentinel/Cluster 运维 | 成熟但复杂 | 单节点替代多节点集群，运维更简 |
| 监控 | 丰富（INFO/内存图谱） | 基础监控够用，相比 Redis 略少 |
| 社区 | 庞大，十五年积累 | 增长迅速，生态相对年轻 |
| Docker 部署 | 官方镜像 + 大量第三方编排 | 官方镜像可用 |
| K8s Operator | 成熟（Spotahome + 官方 Operator） | 社区方案为主 |
| 商业支持 | Redis Ltd. / 云厂商托管 | Dragonfly 商业版 |

如果从运维角度看，Dragonfly 的核心卖点是**用更少的节点承载相同的流量**——这意味着服务器数量减少一半甚至更多，集群复杂度降低。

## 七、技术选型建议

### 建议选 Dragonfly 的场景

- 高并发缓存（QPS > 500K，且 CPU 有富余核心）
- 大规模 key 存储（内存敏感，追求更低的单位成本）
- 希望简化集群架构（用 2-3 台替代 8-12 台）
- 新项目，无历史包袱

### 建议继续用 Redis 的场景

- 延迟抖动敏感的业务（金融、实时控制）
- 重度依赖 Redis Stack / 模块
- 复杂的 Redis Cluster 分片逻辑迁移成本高
- 需要成熟的运维工具链

### 混合架构

这不是一个非此即彼的选择。可以考虑**分治**：

```
读写频繁的缓存层 → Dragonfly（高吞吐、省内存）
需要 Lua 事务一致性的关键路径 → Redis（确定性好）
热数据流处理 → Dragonfly（Stream 兼容）
历史/Restore 备份 → Redis（兼容生态工具）
```

实际生产案例中，不少团队采用 **Dragonfly 作为上层缓存、Redis 作为降级后备** 的双层策略，兼顾吞吐和稳定性。

## 八、总结

| 维度 | Redis | Dragonfly |
|------|-------|-----------|
| 架构 | 单线程 + 事件循环 | 多线程 + 无锁分片 |
| 多核利用率 | 低 | 高（线性扩展） |
| 单节点吞吐 | 受限于单核 | 可跑满所有核心 |
| 内存效率 | 基准 | 高 30-40% |
| 延迟确定性 | 优秀（无竞争） | 优秀（平均更低） |
| 协议兼容 | 标准 | 95%+ 兼容 |
| 生态成熟度 | 极高 | 快速增长中 |
| 运维复杂度 | 集群运维复杂 | 节点更少、运维更简 |

Dragonfly 并非要杀死 Redis，而是弥补 Redis 在多核时代的产品空白。它用更好的工程实现解决了 Redis 最头疼的两个问题——**多核利用率和内存效率**。

在 2026 年的今天，如果你的 Redis 集群中 CPU 利用率极不均匀，或者内存快用光了但流量还没到瓶颈，Dragonfly 绝对值得一试。

> 不是 Redis 不够好，是这个时代需要更好的。Dragonfly 正在成为那个"更好的选择"。
